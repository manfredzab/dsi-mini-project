Overview

The multi-way join algorithm benchmarking problem posed in the Database Systems Implementation, Hilary Term 2013 examination is solved by three separate tools:

  • dsi-mini-project tool, which implements three multi-way join algorithms:
    • Cascading sort-merge join, which uses binary sort-merge joins to join relations,
    • Trie-based sort-merge join, which joins all relations at the same time using trie representations of relations (but requires that relations would be pre-sorted in the join-order), and
    • Leapfrog join, as described by Todd L Veldhuizen in "Leapfrog Triejoin: A Simple, Worst-Case Optimal Join Algorithm", LogicBlox Technical Report LB1201, December 2012.
  • dsi-mini-project-tests tool, which is used to test dsi-mini-project tool by comparing the results that it produces to the ones produced by H2 RDBMS engine, and
  • dsi-mini-project-timing tool, which is used to gather performance data for different algorithms, supported by dsi-mini-project tool.

Usage of "dsi-mini-project", "dsi-mini-project-tests" and "dsi-mini-project-timing" tools

This section describes how to:

  • run the dsi-mini-project tool,
  • compile it from source files,
  • test the results for correctness using dsi-mini-project-tests tool, and
  • measure the performance of different sorting algorithms using dsi-mini-project-timing tool.

Running "dsi-mini-project" tool

To join a set of relations using a particular join query and a particular join algorithm, navigate to the binaries folder of dsi-mini-project tool ("dsi-mini-project/bin") and execute the command:

./dsi-mini-project <algorithm> -query <query file> -database <database file> [-output <output type>], where

  • algorithm sortmerge, sortmergetrie, leapfrog , and
    • sortmerge specifies the cascading sort-merge join algorithm,
    • sortmergetrie specifies the trie-based sort-merge join algorithm, and
    • leapfrog specifies the leapfrog join algorithm,
  • query file specifies the file containing the join query to be executed,
  • database file specifies the file containing the filenames of the files containing the relations, and
  • output type tuples, time , and
    • tuples (default) specifies that the comma-separated tuples from the resulting join relation should be printed to STDOUT,
    • time specifies that the time (in fractions of a second) taken to produce the resulting relation should be printed to STDOUT.

For example, to perform a multi-way leapfrog join on "query2" and "scale3" of "dataset1-uniform", execute the command ./dsi-mini-project leapfrog -query ../data/query2 -database ../data/dataset1-uniform/scale3/databasefile.

Troubleshooting

If you are getting "-bash: ./dsi-mini-project: Permission denied" error message, give the "execution" permission to the dsi-mini-project tool with the command

chmod +x dsi-mini-project

and re-run the dsi-mini-project executable.

Compiling "dsi-mini-project" tool

To compile the dsi-mini-project tool, execute the following command from the "dsi-mini-project/bin" folder:

make

The dsi-mini-project executable should be produced.

Testing "dsi-mini-project" tool

To test the correctness of the relations produced by dsi-mini-project, dsi-mini-project-tests tool can be used.

The dsi-mini-project-tests tool uses the embedded H2 RDBMS engine (which supports a subset of SQL standard) to execute given queries on given datasets, and then compares the resulting relations with the ones produced by the dsi-mini-project tool.

Running "dsi-mini-project-tests" tool

To run the tests on both datasets ("dataset1-uniform" and "dataset2-zipf"), all queries ("query1" to "query3"), all scales ("scale1" to "scale6") and all algorithms ("sortmerge", "sortmergetrie", "leapfrog"), execute the following command from the binaries folder of the dsi-mini-project-tests tool ("dsi-mini-project-tests/bin"):

java -jar dsi-mini-project-tests.jar <path to the dataset> <path to dsi-mini-project executable>,

for example java -jar dsi-mini-project-tests.jar ../../dsi-mini-project/data/ ../../dsi-mini-project/bin/dsi-mini-project.

Further testing

Memory management

The dsi-mini-project tool has been tested using Valgrind's Memcheck tool, to ensure that it does not:

  • access memory which it shouldn't (areas not yet allocated, areas that have been freed, areas past the end of heap blocks, inaccessible areas of the stack),
  • use uninitialised values,
  • leak memory,
  • do bad frees of heap blocks (double frees, mismatched frees),
  • pass overlapping source and destination memory blocks to memcpy() and related functions.

Memcheck tool reported no problems on all algorithms (cascading sort-merge join, trie-based sort-merge join and leapfrog join).

Call timing

The "hot-path" call graph analysis of the dsi-mini-project tool has been performend using Valgrind's Callgrind tool and visualized using KCacheGrind tool, to ensure that the implementation was not bottle-necked by I/O operations and the actual algorithm performance was being measured.

Timing the performance of "dsi-mini-project" tool

The dsi-mini-project-timing tool can be used to compare the performance of all three multi-way join algorithms (cascading sort-merge join, trie-based sort-merge join and leapfrog join) on all datasets ("dataset1-uniform" and "dataset2-zipf"), all queries ("query1" to "query3") and all scales ("scale1" to "scale6").

For each algorithm, dataset, query and scale configuration, dsi-mini-project-timing tool reports an arithmetic average of elapsed time in three timing runs (running dsi-mini-project with "-output time" flag), in fractions of a second.

Running "dsi-mini-project-timing" tool

To gather the running time averages of all three algorithms on all datasets, queries and scales, navigate to the binaries folder of dsi-mini-project-timing tool ("dsi-mini-project-timing/bin") and execute the following command:

java -jar dsi-mini-project-timing.jar <path to the dataset> <path to dsi-mini-project executable>,

for example java -jar dsi-mini-project-timing.jar ../../dsi-mini-project/data/ ../../dsi-mini-project/bin/dsi-mini-project.
